   # Cross‑Plane Map
   *A structural overview of how the nine UMA planes relate, constrain, and support one another.*
# 1. Overview
The Cross‑Plane Map provides a structural overview of how the nine UMA planes relate, constrain, and support one another.  
It defines the architecture‑level relationships that maintain coherence across the system.

This map is non‑interpretive and non‑narrative.  
It describes structure, not meaning.

---

# 2. Plane List (Canonical Order)

1. Boundary Layer  
2. Witness Literature Support Plane  
3. Documentation Plane  
4. Structural Plane  
5. Governance Plane  
6. Developmental Plane  
7. Dynamic Plane  
8. Integrative Plane  
9. (External) Witness Literature — referenced but not part of UMA’s internal architecture

---

# 3. Cross‑Plane Relationship Map

## 3.1 Boundary Layer → All Planes
The Boundary Layer:
- defines global constraints  
- prevents collapse  
- establishes non‑interpretation  
- sets system‑wide invariants  

All planes must comply with the Boundary Layer.

---

## 3.2 Documentation Plane ↔ Structural Plane
Documentation Plane:
- records structures  
- preserves version lineage  
- maintains referential stability  

Structural Plane:
- defines the objects being documented  
- provides the ontology  

They form a bidirectional relationship:
- structure informs documentation  
- documentation stabilizes structure  

---

## 3.3 Governance Plane → All Planes
Governance Plane:
- enforces constraints  
- defines prohibitions  
- ensures non‑coercion  
- prevents domain collapse  

Governance constrains:
- structure  
- documentation  
- development  
- dynamics  
- integration  

No plane may override governance.

---

## 3.4 Developmental Plane → Structural + Dynamic
Developmental Plane:
- defines sequencing  
- controls emergence  
- prevents premature capability  

It governs:
- when structures appear  
- when dynamic behavior activates  

Structure defines *what* exists.  
Development defines *when* it exists
